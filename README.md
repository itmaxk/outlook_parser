# Outlook Parser

Утилита для мониторинга входящих писем Outlook 365 (локальный клиент Windows) и автоматического выполнения HTTP-запросов по настраиваемым правилам.

**Пример использования:** при получении письма с темой «Sonar 42» автоматически отправляется GET-запрос `http://localhost:3004/process/42?projectPath=rgs-life/implementation`.

## Требования

- Windows 10/11
- Python 3.10+
- Microsoft Outlook (десктопный клиент), запущенный и авторизованный
- Outlook должен быть открыт во время работы утилиты

## Установка

```bash
cd D:/Projects/outlook_parser
pip install -r requirements.txt
```

Зависимости:
- **FastAPI** + **uvicorn** — веб-сервер и API
- **SQLAlchemy** — ORM для SQLite
- **Pydantic** — валидация данных
- **httpx** — асинхронные HTTP-запросы
- **pywin32** — взаимодействие с Outlook через COM

## Запуск

```bash
cd D:/Projects/outlook_parser
python main.py
```

После запуска:
1. В консоли появится сообщение `Uvicorn running on http://127.0.0.1:8765`
2. Откройте в браузере **http://localhost:8765**
3. База данных SQLite создастся автоматически в папке `data/`
4. Подключение к Outlook произойдёт в фоне — если Outlook не запущен, утилита будет повторять попытки каждые 30 секунд

Для остановки нажмите `Ctrl+C` в консоли.

### Настройка порта

Отредактируйте файл `config.py`:

```python
HOST = "127.0.0.1"
PORT = 8765          # измените на нужный порт
LOG_LEVEL = "info"   # debug / info / warning / error
```

## Веб-интерфейс

Интерфейс доступен по адресу **http://localhost:8765** и состоит из двух вкладок.

### Управление правилами

Вкладка **«Правила»** — таблица всех созданных правил.

#### Создание правила

1. Нажмите кнопку **«+ Новое правило»**
2. Заполните поля:
   - **Имя** — понятное название правила (например, «Sonar — запуск процесса»)
   - **Приоритет** — число; правила с большим приоритетом проверяются первыми. Если несколько правил совпадают, все они будут выполнены
   - **Активно** — включено/выключено правило
3. Добавьте **условия** (кнопка «+ Условие»):
   - Выберите **поле** письма (subject, body, sender и др.)
   - Выберите **оператор** (contains, pattern, regex и др.)
   - Введите **значение** или шаблон
   - Флаг **Aa** — учитывать регистр
   - Все условия в одном правиле объединяются через **AND** — все должны совпасть
4. Укажите **действие**:
   - **URL** — адрес для HTTP-запроса. Поддерживает переменные в фигурных скобках: `http://example.com/process/{ID}`
   - **Метод** — GET, POST, PUT, PATCH, DELETE
   - **Тело запроса** — необязательно, для POST/PUT. Также поддерживает переменные: `{"id": "{ID}"}`
5. Нажмите **«Сохранить»**

#### Редактирование и удаление

- Нажмите **карандаш** (✎) рядом с правилом для редактирования
- Нажмите **корзину** (🗑) для удаления (с подтверждением)

### Просмотр логов

Вкладка **«Логи»** показывает журнал обработки входящих писем.

Для каждого письма отображается:
- **Время** получения
- **Тема** и **отправитель**
- **Правило**, которое сработало (или «-» если ни одно не подошло)
- **Совпало** — да/нет
- **URL** — фактический URL выполненного запроса (с подставленными переменными)
- **HTTP** — код ответа (200, 404 и т.д.)
- **Ошибка** — текст ошибки, если запрос не удался

**Функции:**
- **Фильтр** — показать только совпавшие / несовпавшие письма
- **Обновить** — загрузить свежие логи
- **Очистить логи** — удалить все записи из журнала
- Логи **обновляются автоматически** каждые 10 секунд, пока вкладка активна

## Система правил

### Поля для сопоставления

| Поле | Описание | Пример значения |
|------|----------|-----------------|
| `subject` | Тема письма | `Sonar 42` |
| `body` | Текст письма | `Build failed for project...` |
| `sender` | Email отправителя | `jenkins@company.com` |
| `to` | Получатели (To) | `dev-team@company.com` |
| `cc` | Копия (CC) | `manager@company.com` |
| `importance` | Важность | `low`, `normal`, `high` |
| `categories` | Категории Outlook | `Работа, Срочное` |

### Операторы

| Оператор | Описание | Извлекает переменные? |
|----------|----------|-----------------------|
| `contains` | Поле содержит подстроку | Нет |
| `not_contains` | Поле не содержит подстроку | Нет |
| `equals` | Точное совпадение | Нет |
| `starts_with` | Поле начинается с подстроки | Нет |
| `ends_with` | Поле заканчивается подстрокой | Нет |
| `regex` | Регулярное выражение (Python `re`) | Да — именованные группы `(?P<NAME>...)` |
| `pattern` | Шаблон с плейсхолдерами `{NAME}` | Да — каждый `{NAME}` становится переменной |

### Шаблоны и переменные

Операторы `pattern` и `regex` умеют извлекать переменные из текста письма. Эти переменные затем подставляются в URL и тело HTTP-запроса.

**Оператор `pattern`** — простой синтаксис с `{ИМЯ_ПЕРЕМЕННОЙ}`:

```
Шаблон:    Sonar {ID}
Тема:      Sonar 42
Результат: ID = "42"
```

```
Шаблон:    Build {STATUS} for {PROJECT}
Тема:      Build failed for payment-service
Результат: STATUS = "failed", PROJECT = "payment-service"
```

Под капотом `pattern` компилируется в регулярное выражение: `Sonar {ID}` → `^Sonar (?P<ID>.+?)$`

**Оператор `regex`** — полноценные регулярные выражения Python:

```
Regex:     Ticket #(?P<NUM>\d+):\s+(?P<TITLE>.+)
Тема:      Ticket #789: Fix login page
Результат: NUM = "789", TITLE = "Fix login page"
```

### Примеры правил

#### 1. Запуск процесса по номеру из Sonar

- **Условие:** field=`subject`, operator=`pattern`, value=`Sonar {ID}`
- **URL:** `http://localhost:3004/process/{ID}?projectPath=rgs-life/implementation`
- **Метод:** GET
- **Результат:** письмо «Sonar 42» → GET `http://localhost:3004/process/42?projectPath=rgs-life/implementation`

#### 2. Уведомление о сборке от Jenkins

- **Условие 1:** field=`sender`, operator=`contains`, value=`jenkins`
- **Условие 2:** field=`subject`, operator=`regex`, value=`(?P<PROJECT>[\w-]+)\s+#(?P<BUILD>\d+)\s+(?P<STATUS>\w+)`
- **URL:** `http://localhost:9090/webhook/build?project={PROJECT}&build={BUILD}&status={STATUS}`
- **Метод:** POST
- **Результат:** письмо от jenkins с темой «payment-api #156 SUCCESS» → POST с параметрами `project=payment-api`, `build=156`, `status=SUCCESS`

#### 3. Пересылка писем определённой категории

- **Условие:** field=`categories`, operator=`contains`, value=`Мониторинг`
- **URL:** `http://localhost:5000/api/alerts`
- **Метод:** POST
- **Тело:** `{"subject": "{SUBJECT}"}` *(примечание: встроенная переменная SUBJECT не поддерживается — используйте regex/pattern для извлечения нужных данных)*

#### 4. Фильтр по важности

- **Условие 1:** field=`importance`, operator=`equals`, value=`high`
- **Условие 2:** field=`subject`, operator=`starts_with`, value=`[ALERT]`
- **URL:** `http://localhost:8080/urgent-handler`
- **Метод:** GET

## API

Все эндпоинты доступны по адресу `http://localhost:8765`.

### Правила

| Метод | URL | Описание |
|-------|-----|----------|
| GET | `/api/rules` | Список всех правил. Параметр `?enabled=true/false` для фильтрации |
| POST | `/api/rules` | Создать правило |
| GET | `/api/rules/{id}` | Получить правило по ID |
| PUT | `/api/rules/{id}` | Полное обновление правила |
| PATCH | `/api/rules/{id}` | Частичное обновление |
| DELETE | `/api/rules/{id}` | Удалить правило |

**Пример создания правила через curl:**

```bash
curl -X POST http://localhost:8765/api/rules \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Sonar trigger",
    "priority": 10,
    "enabled": true,
    "conditions": [
      {"field": "subject", "operator": "pattern", "value": "Sonar {ID}", "case_sensitive": false}
    ],
    "action_url": "http://localhost:3004/process/{ID}?projectPath=rgs-life/implementation",
    "action_method": "GET"
  }'
```

### Логи

| Метод | URL | Описание |
|-------|-----|----------|
| GET | `/api/logs` | Список логов. Параметры: `?matched=true/false`, `?rule_id=N`, `?limit=100`, `?offset=0` |
| DELETE | `/api/logs` | Удалить все логи |

## Архитектура

Приложение работает в двух потоках внутри одного процесса:

```
main.py
│
├── Поток 1 (основной): uvicorn → FastAPI
│   ├── API эндпоинты (CRUD правил, логи)
│   ├── Статические файлы (веб-интерфейс)
│   └── SQLAlchemy → SQLite
│
└── Поток 2 (daemon): Outlook COM Watcher
    ├── pythoncom.CoInitialize()
    ├── DispatchWithEvents("Outlook.Application")
    ├── OnNewMailEx → очередь
    └── Цикл (0.5с):
        ├── PumpWaitingMessages() — обработка COM-событий
        └── Обработка очереди:
            ├── Получение MailItem по EntryID
            ├── Извлечение данных (тема, отправитель и т.д.)
            ├── Сопоставление с правилами
            ├── Выполнение HTTP-запросов
            └── Запись в лог
```

**Ключевые решения:**
- `sys.coinit_flags = 0` (STA) устанавливается до любых COM-импортов
- SQLite используется с `check_same_thread=False` для доступа из обоих потоков
- Ссылка на COM-обработчик хранится на уровне модуля для защиты от сборщика мусора
- При потере соединения с Outlook — автоматическое переподключение каждые 30 секунд

## Решение проблем

### Outlook не подключается

- Убедитесь, что Outlook запущен и авторизован (не на экране ввода пароля)
- Проверьте логи в консоли — должно быть сообщение `Connected to Outlook`
- Если в логах `Cannot connect to Outlook, retrying in 30s...` — Outlook недоступен через COM

### Письма не обрабатываются

- Проверьте, что хотя бы одно правило создано и **включено** (колонка «Активно» = «Да»)
- Откройте вкладку «Логи» — там должна появиться запись для каждого входящего письма
- Если запись есть, но `Совпало = Нет` — проверьте условия правила
- Установите `LOG_LEVEL = "debug"` в `config.py` для подробного вывода в консоль

### HTTP-запрос не выполняется

- Проверьте колонку «Ошибка» в логах
- Типичные проблемы:
  - Целевой сервер не запущен → `Connection refused`
  - Неправильный URL → `404`
  - Таймаут → запрос занимает более 30 секунд

### Переменные не подставляются

- Убедитесь, что оператор — `pattern` или `regex`
- Для `pattern`: имена переменных в условии и URL должны совпадать (`{ID}` в обоих местах)
- Для `regex`: используйте именованные группы `(?P<NAME>...)` — только они становятся переменными
- Операторы `contains`, `equals` и другие не извлекают переменные
